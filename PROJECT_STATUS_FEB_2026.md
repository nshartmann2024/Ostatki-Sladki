# PROJECT STATUS — FEB 2026 (UPDATED)

## 1. Архитектура
- Core независим от платформ (логика в `core/core_engine.py`).
- Telegram и Web — тонкие adapters; вызывают `handle_event()`, отображают `ResponsePlan`.
- Контракт ResponsePlan не нарушен (messages, actions, state_patch, ui_mode, ui_hints).
- SQLite — основная БД (`db/app.db`).
- Канонические ингредиенты и алиасы — `data/ingredients_canonical.json`. Seed рецептов — `data/recipes_seed_min.json`, импорт через `db/init_db.py`.

## 2. Реализовано
- **Поиск:** по одному каноническому ингредиенту; `normalize_input()` с токенизацией фразы (первый подходящий токен); canonical + aliases; поиск по `must_have_json`.
- **Уточняющее сообщение:** при успешном распознавании: «Я понял: ищу рецепты с «…».» (Web и Telegram).
- **Web (Streamlit):** карточка рецепта через маркер `#os-recipe-card-marker` и CSS `:has()`; белая карточка с тенью; скрыты fullscreen/toolbar у изображения; `st.image(..., width="stretch")`; улучшена видимость строки поиска (белый фон, граница, скругление); подсказка: «Введите 1 ингредиент (можно фразой). Например: «отварной картофель», «творог», «белый хлеб», «шоколад».».
- **Telegram:** та же логика поиска через core; подсказка при /start и в заменах технических фраз обновлена на тот же текст.
- **БД:** поиск по `must_have_json`; пагинация «Ещё варианты»; похожие рецепты (related) по пересечению ингредиентов.

## 3. Поиск (текущее поведение)
- Поиск по **одному** ингредиенту (первый распознанный из фразы).
- Поддержка фраз: «отварной картофель» → «картофель», «картофель отварной» → «картофель» (токенизация, пунктуация → пробел).
- Справочники: `ingredients_canonical.json` (canonical + aliases); полное совпадение строки, затем перебор токенов.
- Ограничения MVP: нет морфологии, нет стоп-слов, нет поиска по двум ингредиентам (AND/OR), нет ранжирования по релевантности.

## 4. UI Streamlit
- **Карточка рецепта:** не обёртка `<div class="os-card">`, а невидимый маркер `<span id="os-recipe-card-marker">` + селектор `div[data-testid="stVerticalBlock"]:has(#os-recipe-card-marker)` (белый фон, тень, скругление, отступы). Нет пустой «плашки» над фото.
- **Изображение:** скрыты `[data-testid="stImageToolbar"]` и кнопки fullscreen у `st.image`; скругление фото внутри карточки; `width="stretch"`.
- **Строка поиска:** `[data-testid="stTextInput"] input` — фон #FFFFFF, граница rgba(0,0,0,0.15), border-radius 10px; placeholder затемнён. Focus/outline не переопределяются.
- **Разделители:** `hr`, `[data-testid="stDivider"]` скрыты. Адаптивный размер заголовка рецепта (clamp), компактные списки ингредиентов/шагов.
- Ограничения: стили только через CSS, без изменения контракта core; поле поиска и кнопки навигации не внутри карточки.

## 5. Telegram
- Бот запускается без ошибок (`python adapters/telegram/bot.py` при заданном `OS_TG_TOKEN`).
- Использует ту же логику поиска через `handle_event()`; state в памяти (user_key → dict).
- Подсказка при /start и при замене технических фраз: «Введите 1 ингредиент (можно фразой). Например: «отварной картофель», «творог», «белый хлеб», «шоколад».».
- Ограничения: state не персистентный; нет отдельной морфологии/поиска на стороне бота.

## 6. Текущие ограничения
- Поиск только по одному ингредиенту (первый распознанный токен).
- Нет морфологии (стемминг, нормализация ё/е).
- Нет стоп-слов при токенизации.
- Нет комбинированного поиска (AND/OR по нескольким ингредиентам).
- Нет расширенного ранжирования (только вхождение в `must_have_json`).
- Telegram: state в памяти, при перезапуске бота теряется.

## 7. Следующие шаги
1. При необходимости — добавить стоп-слова или алиасы для частых фраз («отварной», «вареный» и т.п.).
2. Рассмотреть поиск по двум ингредиентам (AND) как следующий шаг после MVP.
3. При росте числа рецептов — индексация или кэш по canonical для ускорения поиска.
4. Опционально: персистентный state для Telegram (БД или Redis).

## 10. PHOTO SYSTEM
- **resolve_photo_path(recipe_id):** единый резолвер путей к фото рецептов в Web; возвращает абсолютный путь к файлу или `None`. Логика: 1) явный маппинг `EXPLICIT_MAP` (например `erdapfelgulasch_v1` → `erdaepfelgulasch.jpg`), 2) прямое совпадение `recipe_id.jpg`, 3) для id с суффиксом `_v1` — попытка `{base_id}.jpg`. Проверка существования файла через `Path.exists()`.
- **PROJECT_ROOT:** вычисляется как `Path(__file__).resolve().parents[2]` (от `app.py` вверх: web → adapters → корень проекта). **PHOTO_DIR = PROJECT_ROOT / "assets" / "photos"** — не зависит от текущей рабочей директории; корректно работает при запуске из любой папки и в Streamlit Cloud.

## 11. SEARCH UPDATE
- **Canonical:** список канонических ингредиентов в `data/ingredients_canonical.json` (мясо, колбаса, бекон, грудинка, творог, картофель и др.).
- **Aliases:** мясные алиасы (колбаса, ветчина, бекон, сосиски, сардельки, грудинка → мясо); формы слов и синонимы (картошка, лука, чеснока, бананы, яблоки, батон, булка и т.д.). В Web перед вызовом core применяется словарь `NORMALIZE_WORDS` для частых форм.
- **BASIC_EXCLUDE:** в скриптах аудита/обновления тегов используется список «базовых» ингредиентов, которые не добавляются в `must_have_json` массово (масло, яйцо, лук и т.д.), с исключениями для рецептов, где ингредиент ключевой (лук в луковом супе, молоко в десертах).

## 12. STREAMLIT CLOUD READY
- Пути к ресурсам не зависят от cwd: фото через **PROJECT_ROOT** и **PHOTO_DIR**, БД через `_root` или путь от `__file__`.
- **assets/photos** — все изображения рецептов в одной папке; резолвер находит файлы по `recipe_id` и явному маппингу.

- Относительные пути от корня проекта используются последовательно; `st.image` и работа с файлами проверяют существование перед отображением.
